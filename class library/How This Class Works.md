# How This Class Works

## The Goal
This class runs like an **engineering studio**. Instead of large assignments at the end of the semester, teams ship **small improvements every week**.

Your goal is to build a **working project that can be demonstrated reliably** by the end of the semester.

The focus is not building something huge.

The focus is building something **that actually works.**

---

# The Three Rules

## 1. GitHub is the Source of Truth

All work must appear in your **team GitHub repository**.

This includes:

- tasks (Issues)
- planning (Project board)
- code
- documentation
- evidence of work

If something is not in GitHub, it does **not count as completed work**.

---

## 2. Evidence Matters

Every student must show proof of their contribution.

Examples of evidence:

- Pull Requests
- commits
- Issues closed
- review comments
- documentation updates
- screenshots of working features

Rule:

**If it isn't linked, it didn't happen.**

---

## 3. We Work in Weekly Sprints

Work happens in **short cycles called sprints**.

Each sprint ends with a team submission called the **Weekly Sprint Packet**.

This keeps teams moving forward and prevents last‑minute work at the end of the semester.

---

# Weekly Rhythm

Each week follows the same structure.

## Wednesday Class

We will:

- review progress
- fix blockers
- learn one small engineering concept
- work on the project during studio time

Most real progress should happen **during class time**.

---

## Next Wednesday Check

Teams should arrive with:

- partial progress
- questions
- blockers
- demo progress

The instructor will check:

- project scope
- GitHub workflow
- evidence quality

---

## Friday Deadline

The **Weekly Sprint Packet** is due.

The Sprint Packet contains:

- demo link
- board snapshot
- sprint notes
- individual contribution receipts

---

# The Weekly Workflow

Most work follows this pattern:

Issue → Branch → Pull Request → Merge → Demo

This process creates a record of your work.

---

# What You Submit Every Week

Each team submits **one Sprint Packet Issue**.

Inside the Sprint Packet:

- demo link or video
- board snapshot
- sprint notes
- next sprint plan

Each student must then **add a comment with receipts**.

Minimum:

2–3 links showing your work.

---

# Week 1 Special Rule

Week 1 setup work is **required but not graded**.

This includes:

- creating the repo
- writing the team agreement
- pitching a project idea
- creating the first sprint packet

These documents will evolve throughout the semester.

---

# What Makes a Strong Project

Good projects are:

- demoable
- feasible
- clearly scoped

Bad projects are:

- too large
- unclear
- impossible to demonstrate

---

# The Goal of This Class

By the end of the semester your team should have:

- a working project
- a reliable demo
- clear documentation
- visible evidence of your contributions

The final result should show **steady progress across the semester**, not a rushed project at the end.
