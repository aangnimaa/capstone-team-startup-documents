# POLICY — Evidence & Receipts

## Purpose

Capstone Design uses an **evidence‑based grading system**.

Instead of guessing who contributed to the project, we rely on **verifiable links in GitHub**.

This makes grading fair and keeps teams accountable.

---

# The Core Rule

If it isn't linked, it didn't happen.

Work must be visible in GitHub.

---

# What Are Receipts?

Receipts are **links that show the work you did**.

Each student must post receipts in the **Weekly Sprint Packet issue**.

Minimum per week:

2–3 links.

---

# Examples of Valid Receipts

Acceptable evidence includes:

- Pull Requests
- commits
- Issues created
- Issues closed
- review comments
- documentation updates
- CI test runs
- screenshots of working features

---

# Examples

Example receipt comment:

```
Contribution Receipts

PR
https://github.com/team/repo/pull/21

Issue completed
https://github.com/team/repo/issues/15

Review comment
https://github.com/team/repo/pull/20#discussion
```

---

# What Does NOT Count

These do not count as evidence:

- saying you worked without proof
- group work without individual receipts
- work done outside the repository
- verbal explanations without links

---

# Where Receipts Are Posted

Receipts must be posted as a **comment on the Weekly Sprint Packet issue**.

Do not post them elsewhere.

---

# Why This Policy Exists

This policy helps:

- ensure fair grading
- track individual contributions
- identify problems early
- build professional engineering habits

---

# Common Mistakes

Students often make these mistakes:

Working without an Issue

Forgetting to link the Pull Request

Posting screenshots without links

Waiting until Friday to do all work

Avoid these problems by updating GitHub throughout the week.
